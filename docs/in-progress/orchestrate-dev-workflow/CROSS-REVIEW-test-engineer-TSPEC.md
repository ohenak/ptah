## Questions

| ID | Question |
|----|---------|
| Q-01 | Section 6.4 contains two algorithm blocks that contradict each other (the original and the "corrected" version). Which is the canonical implementation? The second block (the explicit correction) appears to be correct, but the first block should be removed or struck through to prevent confusion. |
| Q-02 | The `pollUntilTerminal()` algorithm (Section 6.5) accesses `state.workflowConfig` — is `workflowConfig` a field on `FeatureWorkflowState` returned by `queryWorkflowState()`? If not, where does the `emitProgressLines()` function get the phase label and title? The `PHASE_LABELS` static map suggests it is a hardcoded lookup, but the algorithm references `state.workflowConfig.phases.find(...)`. This needs clarification before a test can be written for phase label emission. |
| Q-03 | For `checkArtifactExists`, the path is constructed as `${featurePath}${docType}-${slug}.md`. When `featurePath` does NOT end with `/` (e.g., `featurePath = "docs/in-progress/my-feature"`), the constructed path would be `docs/in-progress/my-featureREQ-my-feature.md` — which is incorrect. Should there be a path normalization step, and should this be tested? |
| Q-04 | The `runReviewCycle()` revision context assembly (Section 5.9) says missing cross-review files are silently skipped by `DefaultContextAssembler`. But the TSPEC test table (Section 13.2) for `feature-lifecycle.test.ts` does not include a test for this silent-skip behavior at the workflow level. Is the expectation that this is tested at the `DefaultContextAssembler` unit test level only, or should there be a test in `feature-lifecycle.test.ts` that verifies the context list is built correctly even when some paths do not exist? |
| Q-05 | Section 5.2 specifies `isCompletionReady()` passes `{ discipline: "default" } as FeatureConfig` as a stub — this casts an incomplete object to `FeatureConfig`. If `FeatureConfig` gains new required fields in the future, this cast will silently compile but `computeReviewerList()` may behave incorrectly. Should a typed minimal fixture be used instead of a cast? |

## Positive Observations

- The TSPEC is exceptionally detailed: every function has an exact signature, algorithm pseudocode, and co-change requirements called out. This level of precision makes the test-writing task straightforward for all core behaviors.
- The `FakeTemporalClient` extension in Section 5.11 uses a call-recording pattern (`listWorkflowsByPrefixOptions`, `humanAnswerSignals`) that follows the existing factory conventions in `factories.ts` well. This approach makes call-assertion tests clean.
- The `VALUE_MATCHERS` ordering concern (longer matches before shorter ones to prevent partial matches) is explicitly noted in Section 5.4, which is a testability improvement — it signals what the ordering regression tests should cover.
- The two-step `extractRecommendationValue()` + `parseRecommendation()` refactor (Section 5.4) cleanly separates heading extraction from value matching, making both functions unit-testable in isolation. This is good test design.
- The static-analysis test for `mapRecommendationToStatus()` deletion (Section 12.3) is a concrete, automatable verification that the dead code removal is complete — this pattern is already proven in the existing `feature-lifecycle.test.ts`.
- The error table in Section 12.1 is comprehensive and maps 1:1 to test cases in Section 13.2, making traceability easy to verify.
- Backward-compatibility regression tests for old agent IDs (`eng`, `qa`, `pm`, `fe`) are explicitly listed in Section 13.2 for `agentIdToSkillName` — this is the correct approach for REQ-NF-01 coverage.

## Recommendation

**Need Attention**

> F-01 through F-04 are High severity and require resolution before implementation begins. F-01 (missing negative test for partial-string matching) and F-04 (contradictory algorithm blocks with missing test cases) represent risks that could allow subtle bugs to survive the test suite. F-02 (undefined field access in polling loop) and F-03 (unverified FakeTemporalClient status-filter behavior) represent integration-boundary gaps where the fake doubles do not faithfully represent the real system constraints.
>
> Required actions:
> 1. **F-01:** Add a negative test for `parseRecommendation` with a superset string (e.g., `"Approved with Minor Issues — see findings"`) to verify exact match, not substring match.
> 2. **F-02:** Clarify whether `FeatureWorkflowState` returned by `queryWorkflowState()` includes `workflowConfig`, and add a test for the `emitProgressLines()` fallback path (unknown phase ID).
> 3. **F-03:** Add a test that populates `workflowIds` AND `workflowStatuses` on `FakeTemporalClient` and verifies the status filter correctly excludes non-matching statuses.
> 4. **F-04:** Remove the first (incorrect) algorithm block from Section 6.4, retain only the corrected version, and add tests for the `lastContiguousIndex === -1` case (empty folder), all-artifacts-present case, and a gap at position 2.
