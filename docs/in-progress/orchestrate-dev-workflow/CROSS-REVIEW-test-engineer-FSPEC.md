If `resolveNextPhase` catches it silently, the guard is defeated. The acceptance test AT-WF-01-C tests `evaluateSkipCondition` directly, but there is no acceptance test for what happens when `resolveNextPhase` is called with a phase that has an `artifact.exists` skip_if and the map is not yet populated. Is the `ptah.workflow.yaml` path hard-coded as `./ptah.workflow.yaml` relative to the process working directory, or is it resolved relative to the REQ file path? Does "reviewer statuses" mean the full status object for all reviewers, or only the statuses that have changed since the last tick? ## Positive Observations

- The behavioral flow diagrams in each FSPEC section are detailed enough to derive decision-branch tests directly. Each branch node maps cleanly to a test case.
- BR-CLI-01 explicitly covers both zero-byte and whitespace-only files as "empty," which prevents a common implementation bug. AT-CLI-01-C covers the whitespace case specifically.
- The `revisionCount` clamping rule (BR-CR-02: 0 and negative → treat as 1) is explicit and testable. The versioning examples table in FSPEC-CR-01 includes `0` and `-5` cases, making it straightforward to write parametric unit tests.
- FSPEC-WF-01 correctly identifies the Temporal determinism constraint and specifies the pre-loop Activity pattern as the solution. The throw guard in `evaluateSkipCondition()` when called before the pre-loop is a strong defensive contract that is fully testable.
- The legacy fallback in FSPEC-WF-02 (no `implementation-review` phase → use `signOffs.qa && signOffs.pm`) is explicitly documented with acceptance tests (AT-WF-02-C, AT-WF-02-D), preserving backward-compatibility testability.
- AT-CR-01-E and AT-CR-01-F for `writtenVersions` track both the update-after-dispatch and the ContinueAsNew preservation contracts separately — good test decomposition.
- FSPEC-CR-03's context assembly example table (round/reviewer/status/writtenVersions/files) is precise enough to drive a table-driven unit test directly.

## Recommendation

**Need Attention**

> F-01 through F-04 are High severity findings. F-01 and F-02 represent contradictions or specification gaps that will cause the written acceptance tests to be incorrect or untestable as specified. F-04 makes it impossible to write the `parseRecommendation()` unit tests without guessing the API contract. These must be resolved before the TSPEC can define a testable interface.
