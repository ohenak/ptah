## Findings

| ID | Severity | Finding | Section ref |
|----|----------|---------|------------|
| F-01 | Medium | **`parseRecommendation()` signature change is a breaking change to `cross-review-activity.ts` that is not reflected in `ReadCrossReviewInput`.** FSPEC-CR-02 specifies that heading-extraction logic moves from `parseRecommendation()` to the caller (`readCrossReviewRecommendation`). This means `readCrossReviewRecommendation` must now (a) read the raw file content, (b) run the HEADING_PATTERN / BOLD_PATTERN / TABLE_PATTERN extraction in-line, and (c) call the refactored `parseRecommendation(extractedValue)`. However, `ReadCrossReviewInput` in `temporal/types.ts` has no field for the pre-extracted recommendation value ‚Äî and there is no reason to add one, since extraction happens inside the activity body. The key issue is: the FSPEC states the refactored `parseRecommendation()` takes "the normalized value of the `Recommendation:` field only" but does not specify what "normalized" means in the caller's context. In the current implementation, `parseRecommendation()` normalizes via lowercase internally; after the refactor, does the caller pre-normalize (lowercase) before passing, or does `parseRecommendation()` still lowercase its input? The VALUE_MATCHERS matching loop still calls `normalized.includes(matcher.pattern)` which implies the normalization happens inside the function. The FSPEC behavioral flow says "Normalize: lowercase the recommendation field value" as the first step of `parseRecommendation()`, so normalization stays inside. This is consistent, but the FSPEC must confirm that the caller passes the raw extracted value (before lowercase) so there is no double-normalization risk from the caller pre-lowercasing and the function also lowercasing. | FSPEC-CR-02 / Behavioral Flow |
| F-02 | Medium | **`resolveNextPhase()` signature cascade is unspecified.** FSPEC-WF-01 AT-WF-01-E states that `resolveNextPhase()` must propagate the throw from `evaluateSkipCondition()` when called with an `artifact.exists` condition and `artifactExists` is undefined. For `resolveNextPhase()` to be able to pass `artifactExists` to `evaluateSkipCondition()`, `resolveNextPhase()` must itself accept an `artifactExists` parameter ‚Äî otherwise it cannot forward it. The current signature is `resolveNextPhase(currentPhaseId: string, config: WorkflowConfig, featureConfig: FeatureConfig): string | null` with no `artifactExists` parameter. The FSPEC BR-WF-06 states "All three existing call sites of `evaluateSkipCondition()` in `feature-lifecycle.ts` must pass `state.artifactExists` as the third argument" but those three call sites are inside `resolveNextPhase()` (line 217) and the main loop (lines 1514‚Äì1517, 1529). If `resolveNextPhase()` calls `evaluateSkipCondition()` internally with `artifact.exists` conditions, it needs `artifactExists` passed into it. The FSPEC does not specify the updated `resolveNextPhase()` signature, leaving the implementer to figure out the cascade. This is a High-risk omission because an implementer who only updates `evaluateSkipCondition()` but not `resolveNextPhase()` will get a TypeScript error on the call to `evaluateSkipCondition()` inside `resolveNextPhase()` ‚Äî or they will incorrectly pass `undefined` for `artifactExists`, triggering the throw at runtime. | FSPEC-WF-01 / BR-WF-06 / AT-WF-01-E |
| F-03 | Low | **`isCompletionReady()` call site in `feature-lifecycle.ts` still uses old single-argument form.** FSPEC-WF-02 BR-WF-11 states "All existing call sites of `isCompletionReady()` must be updated to pass `workflowConfig` as the second argument." The current call at line 1642 of `feature-lifecycle.ts` is `isCompletionReady(state.signOffs)`. The `workflowConfig` variable is available in scope as `workflowConfig` (destructured from `params` at line 1363). The FSPEC correctly specifies the function signature change and the call-site update requirement, but does not specify that `state.signOffs` remains the first argument ‚Äî an implementer reading only the FSPEC might incorrectly pass `state.workflowConfig` (the snapshotted config from state) vs the `workflowConfig` from `params`. The FSPEC should confirm the call site should use `isCompletionReady(state.signOffs, workflowConfig)` where `workflowConfig` is the same config passed to `buildInitialWorkflowState()`. In practice both are the same at this call site, but stating it explicitly removes ambiguity. | FSPEC-WF-02 / BR-WF-11 |
| F-04 | Low | **`FakeTemporalClient.listWorkflowsByPrefix` does not currently support `statusFilter`.** The current `FakeTemporalClient` implementation in `factories.ts` (line 1493‚Äì1495) stores workflows keyed by prefix string: `return this.workflowIds.get(prefix) ?? []`. BR-CLI-28 requires `FakeTemporalClient` to "respect the `statusFilter` when filtering its in-memory workflow list so tests can assert the filter is applied." However, the current `workflowIds: Map<string, string[]>` stores only workflow IDs, not statuses. To implement `statusFilter` filtering, `FakeTemporalClient` would need to store `{ workflowId: string; status: string }[]` per prefix, or add a parallel `workflowStatuses: Map<string, string>` map keyed by workflow ID. The FSPEC BR-CLI-28 specifies the filtering requirement but does not specify the data structure change needed in `FakeTemporalClient` to support status-filtered assertions. Implementers must redesign the fake's storage model. The FSPEC should note that the `workflowIds` map must be extended to store `{ id: string; status: string }[]` (or equivalent) to enable status-based filtering assertions. | FSPEC-CLI-01 / BR-CLI-28 |
| F-05 | Low | **`ptah run` config loading ‚Äî `ptah.config.json` vs `ptah.workflow.yaml` distinction is clear but the lenient loader mechanism is unspecified.** FSPEC-CLI-04 BR-CLI-25 states "`ptah run` uses a lenient config loader: absence of a `discord` section in `ptah.config.json` is treated as 'Discord not configured' and is not an error." The current `ConfigLoader` in `loader.ts` (used by `ptah start`) requires `discord` as a non-optional field. BR-CLI-25 correctly specifies the behavioral requirement but does not specify whether this is implemented by: (a) creating a new `LenientConfigLoader` class, (b) adding an optional `{ requireDiscord?: boolean }` parameter to the existing `ConfigLoader`, or (c) using a different interface entirely. This is a non-trivial architectural decision ‚Äî the `TemporalOrchestrator` and `start.ts` command both depend on the current strict `ConfigLoader` signature. An implementer who naively modifies `ConfigLoader` to make `discord` optional would break `ptah start`'s strict validation. The FSPEC should indicate which option is intended to prevent breaking the existing config loading path. | FSPEC-CLI-04 / BR-CLI-25 |
| F-06 | Low | **OQ-03 remains open and its resolution affects `se-implement` dispatch behavior.** The FSPEC section 4 Open Questions lists OQ-03 as still open: "When PROPERTIES file is absent during `properties-tests` phase, what specific text does `se-implement` include in its output to indicate the absence? Should `ptah run` surface this as a warning?" REQ-WF-03 states "the workflow treats the missing context document as a soft warning and proceeds ‚Äî the `se-implement` agent is responsible for noting the absence in its output." This defers the question entirely to the agent, but there is a workflow-level implication: `resolveContextDocuments()` with a missing `{feature}/PROPERTIES` reference will produce a path like `docs/in-progress/my-feature/PROPERTIES-my-feature.md` regardless of whether the file exists. The activity will pass a non-existent file path in `contextDocumentRefs`. The FSPEC and REQ state this is acceptable (soft warning), but FSPEC-WF-01 does not specify what `checkArtifactExists` returns for PROPERTIES when `properties-creation` was skipped, nor does it specify whether the `artifact.exists` skip condition for `properties-tests` should be present. If `properties-tests` has no `skip_if`, it will always run even when PROPERTIES is absent, which is the intended behavior per REQ-WF-03. This is consistent but should be confirmed explicitly: `properties-tests` must NOT have a `skip_if: { field: "artifact.exists", artifact: "PROPERTIES" }` condition in the default YAML, or it would be silently skipped when PROPERTIES is absent. | FSPEC-WF-01 / REQ-WF-03 / OQ-03 |

## Positive Observations

- All four Medium findings from v2 (F-01 through F-04) are addressed with precision and specificity. The breaking-change identification for `parseRecommendation()` (SE v2 F-01), the three-part co-change enumeration for `SkipCondition` (SE v2 F-02), the `FakeTemporalClient.signalHumanAnswer()` co-change (SE v2 F-03), and the `listWorkflowsByPrefix` optional-parameter resolution (SE v2 F-04) are all cleanly specified.
- BR-CLI-28 is the most detailed business rule in the document ‚Äî it specifies the exact optional-parameter shape, the TypeScript union type for status values, the Temporal visibility query clause, and the `FakeTemporalClient` storage and filtering behavior. This level of specificity directly reduces TSPEC authoring time.
- AT-WF-01-H (pre-loop scan exhaustive over full config regardless of `startAtPhase`) is a new acceptance test that closes a subtle correctness gap ‚Äî BR-WF-07 is now testable with a concrete multi-phase fixture.
- AT-CR-02-E's static-analysis test mechanism (source-scan using `fs.readFileSync` to assert `"mapRecommendationToStatus"` token is absent) is the right approach for a deletion acceptance test and reuses existing `feature-lifecycle.test.ts` patterns.
- The AT-CR-01-F test fixture note identifying that the existing `buildContinueAsNewPayload()` unit test fixture must be extended to include `reviewStates` is an implementer-facing detail that prevents a "passing tests, missing behavior" bug.
- The `approved_with_minor_issues` ‚Üí `"approved"` collapse is now end-to-end consistent: BR-CLI-18 (progress reporter), BR-WF-09 (`signOffs` mapping), AT-WF-02-F (acceptance test), all agree.

## Recommendation

**Approved with Minor Issues**

> F-01 and F-02 are Medium severity but do not block implementation ‚Äî they are clarifications on already-specified behavior that can be resolved in the TSPEC:
>
> - **F-01**: Confirm in the TSPEC or in a FSPEC-CR-02 clarification that `readCrossReviewRecommendation` passes the raw (not pre-lowercased) extracted value to `parseRecommendation()`, which then normalizes internally.
> - **F-02**: The TSPEC author will need to specify the updated `resolveNextPhase()` signature to accept `artifactExists?` as a third parameter, or choose an alternative forwarding mechanism. This should be explicitly acknowledged as a TSPEC decision point.
>
> F-03 through F-06 are Low severity and can be handled in TSPEC with no risk to implementation correctness.
>
> The FSPEC is sufficiently complete and consistent for TSPEC authoring to begin.
