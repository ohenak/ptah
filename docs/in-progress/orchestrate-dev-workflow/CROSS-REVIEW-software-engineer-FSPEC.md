| F-01 | High | `deriveDocumentType("properties-tests")` currently strips the `-tests` suffix just like it strips `-creation` and `-review` — but the regex `/-(?:creation\|review\|approved)$/` does NOT match `-tests`. So `deriveDocumentType("properties-tests")` already returns `"PROPERTIES-TESTS"`, not `"PROPERTIES"`. The FSPEC relies on this returning `"PROPERTIES"` (per REQ-WF-03 in-scope bullet and FSPEC Section 5 traceability noting REQ-WF-03 as "Not required"). The function must be explicitly extended to handle the `properties-tests` phase ID. The FSPEC says nothing about how the function handles this case, leaving it ambiguous for the implementer. | FSPEC-WF-01 / REQ-WF-03 |
| F-02 | High | `SkipCondition` in `workflow-config.ts` is typed as `{ field: string; equals: boolean }` — it has no `artifact` property. FSPEC-WF-01 adds a new discriminated union variant `{ field: "artifact.exists"; artifact: string }` but the FSPEC does not specify that the `equals` field is absent (or must be absent) in the `artifact.exists` branch. The YAML loader's `validateStructure()` accepts any `skip_if` object today. The FSPEC must specify: (a) the full TypeScript discriminated union definition, (b) that the validator must reject a `skip_if` block where `field === "artifact.exists"` but `artifact` is absent, and (c) that the legacy `equals`-based branch still requires an `equals` boolean. Without this, engineers may implement an ad-hoc type with silent mismatch. | FSPEC-WF-01 / REQ-WF-05 |
| F-03 | High | Signal name mismatch. The existing codebase uses signal name `"user-answer"` (defined at line 657 of `feature-lifecycle.ts`: `const userAnswerSignal = wf.defineSignal<[UserAnswerSignal]>("user-answer")`), but FSPEC-CLI-04 BR-CLI-17 specifies the signal name as `"humanAnswerSignal"`. These are two different names. The FSPEC must resolve whether the new `ptah run` headless path introduces a *new* signal definition `"humanAnswerSignal"` alongside the existing `"user-answer"` signal, or replaces it — and what the migration impact is for `ptah start` which uses `signalUserAnswer()` on the same workflow. Introducing a second signal for the same semantic purpose without removing the first creates a runtime multiplexing ambiguity. | FSPEC-CLI-04 / BR-CLI-17 |

The FSPEC correctly specifies the desired behavior but does not note this divergence from the current implementation as a breaking change to `runReviewCycle`. Implementers may miss the existing filter at line 1319 (`filter((id) => reviewState.reviewerStatuses[id] === "revision_requested")`). The FSPEC should call out explicitly that this filter must be removed. | FSPEC-CR-03 / BR-CR-10 |

## Questions

| ID | Question |
|----|---------|
| Q-01 | FSPEC-CLI-04 introduces `"humanAnswerSignal"` as the signal name for stdin-based answers, but the existing codebase uses `"user-answer"` for Discord-routed answers. Should both code paths use the same Temporal signal, or are these two separate signals on the same workflow? If separate, how does the workflow handler distinguish which signal to wait on depending on the interaction mode (Discord vs. stdin)? |
| Q-02 | FSPEC-WF-01 specifies that `checkArtifactExists` is called for each phase with `skip_if: { field: "artifact.exists" }` before the main phase loop. With the `startAtPhase` mechanism, if `startAtPhase` is set to `"tspec-creation"`, will the pre-loop activity still check artifact existence for `req-creation`? Does the pre-loop always run for ALL `artifact.exists` phases regardless of `startAtPhase`, or only for phases at or after `startAtPhase`? |
| Q-03 | FSPEC-CLI-03 says the finding count `N` is derived from the `## Findings` table row count in the cross-review file. Does this count include the header row, or only data rows? The phrasing "total row count" is ambiguous — a 4-finding table has 6 rows (header separator, header, separator, 4 data rows). The AT-CLI-03-A example shows `N=4` findings — please confirm the counting rule is "data rows only, not header/separator rows". |
| Q-04 | FSPEC-CR-02 specifies `parseRecommendation()` as the single authoritative parser, but the current `cross-review-activity.ts` already calls `parseRecommendation()`. The function to be deleted is `mapRecommendationToStatus()` in `feature-lifecycle.ts`. Are there any callers of `mapRecommendationToStatus()` outside `feature-lifecycle.ts` that also need to be migrated? |
| Q-05 | FSPEC-CLI-01 Step 5 queries for workflows matching prefix `"ptah-{slug}"` with status filter `["Running", "ContinuedAsNew"]`. Is this prefix query exact-match on the workflow ID `ptah-{slug}`, or a prefix-match that could also return `ptah-{slug}-suffix` variants? Given that `startFeatureWorkflow()` sets the workflow ID to exactly `ptah-{featureSlug}`, the query prefix `"ptah-my-feature"` would also match a hypothetical `ptah-my-feature-v2`. Should the duplicate check use exact ID lookup rather than prefix search? |

## Positive Observations

- The three-tier phase resolution priority (explicit flag → auto-detection → default) is cleanly specified with unambiguous precedence in FSPEC-CLI-02. The contiguity rule for gap handling is precise and implementable without PM clarification.
- FSPEC-WF-01 correctly identifies the Temporal determinism violation risk for filesystem I/O and specifies the pre-loop Activity pattern as the solution. The guard error throw for premature `evaluateSkipCondition` calls is an excellent defensive programming specification.
- FSPEC-CR-02 correctly specifies first-match-wins ordering for `VALUE_MATCHERS` with the full ordered table. The conservative default (unrecognized = `revision_requested`) is appropriate for a gate function.
- FSPEC-CR-03 context assembly with `writtenVersions` enumeration and silent-skip for missing files is well-specified. The context assembly example table (round 2 multi-reviewer scenario) gives implementers a concrete trace to verify against.
- The error message table in FSPEC-CLI-01 is precise and directly testable. Specifying exact stdout message strings with placeholder positions eliminates ambiguity in acceptance testing.
- FSPEC-CR-01 versioning examples table (including the 0 and -5 clamp cases) covers boundary conditions that would otherwise be overlooked.

## Recommendation

**Need Attention**

> F-01 through F-04 are High severity and must be resolved before implementation begins:
>
> - **F-01**: Explicitly specify that `deriveDocumentType("properties-tests")` must return `"PROPERTIES"` and update the regex or add a special-case lookup to the function contract in the FSPEC.
> - **F-02**: Provide the full TypeScript discriminated union definition for `SkipCondition` in FSPEC-WF-01, including validator requirements for each branch.
> - **F-03**: Resolve the `"humanAnswerSignal"` vs `"user-answer"` conflict and specify whether a new signal definition is introduced or the existing one is reused/renamed.
> - **F-04**: Explicitly require `buildContinueAsNewPayload()` to include `reviewStates` (with `writtenVersions`) in its output. The type change alone (REQ-NF-04) is insufficient without specifying the payload builder update.
>
> F-05 and F-06 should be addressed to prevent misimplementation of reviewer filter logic and discipline-key config format.
>
> F-14 (AT-CLI-02-F test case appears to contradict the phase map) should be corrected before implementation to avoid a failing acceptance test from day one.
