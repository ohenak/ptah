| F-01 | High | `evaluateSkipCondition()` only accepts `{ field: string; equals: boolean }` and reads from `FeatureConfig` boolean fields. ## Questions

| ID | Question |
|----|---------|
| Q-01 | REQ-WF-05 specifies `evaluateSkipCondition()` should implement the `artifact.exists` check "by calling the filesystem service". Temporal workflow functions are deterministic and cannot perform I/O. The REQ acknowledges this in R-01 and proposes a `checkArtifactExists` Activity. Should `SkipCondition` be extended with a discriminated union (`{ field: "config.*"; equals: boolean } | { field: "artifact.exists"; artifact: string }`) and should the workflow pre-fetch artifact existence results into `FeatureWorkflowState` before the main loop begins? A concrete type and state shape is needed before FSPEC/TSPEC can proceed. |
| Q-02 | REQ-CR-07 requires all cross-review versions to be passed to the optimizer. What workflow state field tracks which versions have been written? Does `ReviewState` gain a `writtenVersions: Record<string, number>` field (agentId â†’ highest version written), or is the version simply derived from `revisionCount`? Clarifying this before TSPEC authoring will prevent ambiguity. |
| Q-03 | REQ-CLI-05 and REQ-WF-05 both address phase auto-detection from filesystem artifacts. Is the CLI responsible for computing `startAtPhase` and passing it to `featureLifecycleWorkflow`, or does the workflow engine re-derive the start phase internally? The answer determines where the `checkArtifactExists` Activity is called and whether both layers need access to the same artifact-detection logic. |
| Q-04 | The `isCompletionReady()` function hard-codes `qa` and `pm` sign-off keys. With the 8-role roster, which agent IDs should be treated as completion sign-off agents? Should this be configurable in `ptah.workflow.yaml` (e.g., a `completion_signoffs` list), or is the completion promotion model being deprecated in favour of the `done` phase transition? |
| Q-05 | For REQ-CLI-03, what is the polling mechanism for streaming Temporal workflow progress to stdout? Temporal does not push events; is the CLI expected to poll `queryWorkflowState` on an interval, and if so at what frequency and with what deduplication strategy? |

## Positive Observations

- The Risks section (Section 5) is unusually strong for a REQ document â€” R-01 correctly identifies the Temporal determinism boundary problem, R-02 correctly identifies the optional-parameter backward-compatibility approach, and R-03 correctly identifies the agent context injection problem. These show good engineering awareness from the PM.
- The distinction between `--from-phase` (explicit override) and artifact-based auto-detection (REQ-CLI-04 vs REQ-CLI-05) is well thought out and matches a real operational need.
- REQ-NF-01 explicitly calls out backward compatibility, and the choice to make `revisionCount` optional with default 1 (R-02) is the right technical approach.
- REQ-CR-05 and REQ-CR-06 correctly identify the exact strings used in the orchestrate-dev skill cross-review format template, making these requirements directly implementable once F-09 is resolved.
- The requirement count table (Section 6) and the cross-domain dependency structure are well organised and make the scope clear.

## Recommendation

**Need Attention**

> F-01 through F-04 are High severity and block implementation. Before FSPEC authoring can begin, the following must be resolved:
>
> 1. **F-01 / Q-01:** Extend `SkipCondition` to a discriminated union and specify the new `checkArtifactExists` Activity contract and the state field it populates.
> 2. **F-02:** Correct REQ-NF-03 to target `ReadCrossReviewInput` (not `SkillActivityInput`) for the `revisionCount` field, or add a separate REQ for the `ReadCrossReviewInput` change.
> 3. **F-03 / Q-02:** Specify what workflow state tracks written cross-review version numbers so REQ-CR-07 can be implemented without ambiguity.
> 4. **F-04 / Q-03:** Define the architectural boundary between CLI-side phase detection (REQ-CLI-05) and workflow-side skip logic (REQ-WF-05) to prevent divergent results.
