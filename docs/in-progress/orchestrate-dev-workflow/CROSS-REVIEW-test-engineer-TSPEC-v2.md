| F-01 | Medium | The `resolveStartPhase()` test case for "gap at position 2 (TSPEC absent, PLAN present)" (Section 13.2) is described as deriving `fspec-creation` (the first missing artifact's creation phase). However, the algorithm in Section 6.4 only checks `hasGapArtifactBeyondFSPEC` when `lastContiguousIndex === 0` (only REQ is contiguous). When `lastContiguousIndex === 0` and TSPEC is present but FSPEC is absent, the gap is at position 1 (FSPEC), so `hasGapArtifactBeyondFSPEC = DETECTION_SEQUENCE.slice(2).some(e => e.found)` would be `true` (TSPEC is at index 2 and is found). The algorithm would correctly derive `fspec-creation`. But the test description says "gap at position 2" while the gap is actually at position 1 (FSPEC = index 1). The mismatch between the test description ("gap at position 2") and the actual position being tested (FSPEC = index 1, TSPEC = index 2) creates ambiguity for the test implementor: does "gap at position 2" mean "TSPEC (index 2) is absent" or "the gap is two positions in"? The REQ-CLI-05 AC uses the phrasing "FSPEC absent but TSPEC is present" for this scenario — the test should use that phrasing instead of "gap at position 2 (TSPEC absent, PLAN present)", which contradicts the algorithm (TSPEC IS present in this scenario; FSPEC is absent). | Section 6.4, Section 13.2 |
| F-02 | Medium | `countFindingsInCrossReviewFile()` uses `fs.readFileSync` (synchronous) inside a function called from `emitProgressLines()`, which is called from `pollUntilTerminal()` (an async function using `await sleep`). The TSPEC specifies the synchronous variant explicitly (Section 6.6: `content = fs.readFileSync(filePath, "utf-8")`). However, the `FileSystem` interface in the existing codebase is async (uses `fs.readFile()`). Using `readFileSync` from `node:fs` directly inside the activity would bypass the injected `FileSystem` dependency and make the function untestable with `FakeFileSystem`. There is no test case listed that exercises file reading in `countFindingsInCrossReviewFile()` with an injected fake filesystem — the test only specifies "correct data-row count" and "file missing → ?" but does not specify how the file content is injected. If `readFileSync` is used directly (not via the injected `fs` dependency), the test cannot use `FakeFileSystem` and must use a real file or `spyOn(fs, 'readFileSync')`. This is a testability gap: the test strategy depends on how file reading is actually done in this helper, and the TSPEC contradicts the dependency-injection pattern used everywhere else in the codebase. | Section 6.6, Section 13.2 |
| F-03 | Medium | `readOneLine()` (Section 6.7) is specified to close the readline interface after resolving on the first `"line"` or `"close"` event. No test case covers the buffering edge condition: if a `Readable` stream has two pre-loaded lines, the second line must be discarded and not consumed by a subsequent `handleQuestion()` call. Without this test, a bug where the readline interface consumes an extra buffered line before being closed — causing the next `handleQuestion()` call to receive a stale answer — could go undetected. ## Questions

| ID | Question |
|----|---------|
| Q-01 | Section 6.6 specifies `countFindingsInCrossReviewFile()` uses `fs.readFileSync`. Is this intentional (bypassing the injected `FileSystem`)? If so, the test strategy must explain how file content is injected for testing (e.g., pre-populating a real temp directory or spying on the Node.js `fs` module). If the intent is to use the injected `FileSystem`, `readFileSync` should be changed to `await fs.readFile()` and `emitProgressLines` made async — but the current signature is synchronous (`void`, not `Promise<void>`). The testability strategy for this function needs clarification. |
| Q-02 | The `resolveStartPhase()` algorithm hard-codes `derivedPhase = "fspec-creation"` in the `hasGapBeyondFSPEC` branch rather than reading `DETECTION_SEQUENCE[1].nextPhase`. Is this intentional? If so, is there a test that verifies this value matches `DETECTION_SEQUENCE[1].nextPhase` to prevent drift? |
| Q-03 | Section 5.8 specifies the `checkArtifactExists` path as `${featurePath}${docType}-${slug}.md`. When `featurePath` is `"docs/in-progress/my-feature"` (no trailing slash), the result is `"docs/in-progress/my-featureREQ-my-feature.md"`. The pre-loop algorithm (Section 5.8) passes `state.featurePath ?? "./"` — is there a normalization step that ensures `featurePath` ends with `/`? This was raised as Q-03 in iteration 1 and the path construction spec remains unchanged. Should a trailing-slash normalization test be added to `checkArtifactExists` unit tests? |

## Recommendation

**Approved with Minor Issues**

> No High findings remain. The Medium findings (F-01 through F-05) are all pre-existing gaps that iteration 1 partially addressed or deferred — none represent design errors in the v1.1 TSPEC. The most actionable are:
>
> - **F-01 (Medium):** Test description for "gap at position 2" contains a terminology contradiction that will confuse the test implementor — the test description should match REQ-CLI-05 AC phrasing ("FSPEC absent, TSPEC present") rather than "TSPEC absent, PLAN present."
> - **F-02 (Medium):** The `readFileSync` vs `fs.readFile` inconsistency in `countFindingsInCrossReviewFile()` creates a testability hole that should be resolved before implementation begins to avoid retrofitting the test.
> - **F-05 (Medium):** The `checkArtifactExists` activity test table gap should be filled — the activity has clear, testable contracts and dependency injection that make unit testing straightforward.
>
> These issues can be addressed either in a TSPEC v1.2 pass or as implementation-time clarifications captured in a follow-up note to the implementing engineer.
