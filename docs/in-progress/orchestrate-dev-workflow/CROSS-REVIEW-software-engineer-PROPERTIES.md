| F-01 | High | PROP-RRC-05 tests "silently skipped without error" but the TSPEC (Section 5.9) leaves the silence to the `DefaultContextAssembler` — there is no workflow-level check. The property as written implies the workflow itself checks for file existence and skips, but the implementation path silently defers to the skill-activity context assembler. The test cannot be a pure workflow unit test because the silence only manifests inside the activity. The test level says "Unit" (feature-lifecycle.test.ts) but the actual skip behavior belongs to the context assembler, not to `runReviewCycle()`. The test level should be Integration or the property must be split: (a) workflow builds the path list unconditionally, (b) context assembler skips unreadable refs. | PROP-RRC-05 |
| F-02 | High | PROP-CLI-01 is classified Integration but its test is listed under `ptah/tests/unit/commands/run.test.ts`. Integration tests in Ptah live under `ptah/tests/integration/`. A `run.test.ts` file under `tests/unit/` cannot start a real Temporal workflow or make real FS I/O. The happy-path scenario described in PROP-CLI-01 (REQ exists, no duplicate, config present → workflow starts) requires either a real Temporal client or a `FakeTemporalClient`. If using a fake, this is a unit test, not integration. The test level is wrong and will guide the implementer to the wrong test infrastructure. ## Positive Observations

- The domain decomposition is well-structured. 10 domains with clear boundaries map directly to the implementation phases in the PLAN, making it easy to trace a failing test to the responsible phase.
- Negative properties section (Section 14) is complete and covers all critical "must NOT" behaviors. Specifically PROP-SC-04 (throw before pre-loop) and PROP-SC-05 (no catch propagation) are well-specified with exact error messages — this precision is exactly what an implementer needs to write the right assertion.
- The VALUE_MATCHERS properties (PROP-PR-01 through PROP-PR-09) include a regression guard (PROP-PR-04) that explicitly names all pre-existing phrases. This prevents silent removal of existing behavior during the refactor.
- PROP-CLI-11's requirement that the valid phase list "must not be truncated with `...`" is a precise, testable boundary condition that is easy to forget and hard to discover otherwise.
- Static-structure tests (PROP-PR-09, PROP-CNP-05) are used deliberately for invariants that the type system cannot enforce. This is architecturally sound for the `mapRecommendationToStatus()` removal and the worker registration check.
- The Coverage Matrix (Section 13) reports "None" for all requirement gaps, and cross-checking confirms correct tracing.

## Recommendation

**Need Attention**

Four High or Medium findings require changes before implementation begins:

- **F-01 (High):** PROP-RRC-05 test level is wrong — the silent-skip behavior belongs to the context assembler, not `runReviewCycle()`. Split into two properties with correct test levels or update the statement to accurately describe the workflow layer's contract.
- **F-02 (High):** PROP-CLI-01 test level (Integration) conflicts with its test file placement under `tests/unit/`. Clarify whether FakeTemporalClient drives this test (Unit) or a real cluster is required (Integration in `tests/integration/`).
- **F-03 (High):** Missing negative property for PROP-SC-07: no test verifies that a non-zero `startAtPhase` still triggers the full pre-loop artifact scan. Add a property explicitly covering this invariant.
- **F-04, F-07, F-08 (Medium):** Minor precision fixes needed on duplicate-workflow status enumeration, deduplication key scope, and optimizer context assembly test boundary.
