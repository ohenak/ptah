# Cross-Review: Software Engineer — REQ-message-acknowledgement

**Reviewer Role:** Senior Software Engineer
**Document Reviewed:** REQ-message-acknowledgement.md
**Review Date:** 2026-04-21
**Recommendation:** Need Attention

---

## Summary

The requirements are well-structured and the two-tier acknowledgement approach is sound. However, there is a high-severity defect: the workflow ID format specified in REQ-MA-02 (`ptah-feature-{slug}-{n}`) does not match what the codebase actually generates (`ptah-{slug}`), which will produce incorrect message text at runtime. Several medium-severity gaps exist around error path coverage, reply semantics, and NF wording that must be resolved before implementation to avoid ambiguity in the FSPEC and TSPEC.

---

## Findings

### Finding 1: Workflow ID format in REQ-MA-02 does not match implementation
- **Severity:** High
- **Location:** REQ-MA-02 (Acceptance Criteria)
- **Issue:** The acceptance criteria states the reply must read `Workflow started: ptah-feature-{slug}-{n}`, implying the format `ptah-feature-{slug}-{n}`. The actual workflow ID generated by `TemporalClientWrapperImpl.startFeatureWorkflow()` (line 81, `ptah/src/temporal/client.ts`) is `ptah-{slug}` — no `feature-` infix and no `-{n}` counter. The acceptance criteria will fail for any real workflow.
- **Recommendation:** Correct the example workflow ID in the acceptance criteria to `ptah-{slug}` (e.g. `Workflow started: ptah-017-message-acknowledgement`). If a `-{n}` counter is ever introduced, that is a separate feature and should not be cited here.

---

### Finding 2: Temporal query failure path is excluded from acknowledgement scope
- **Severity:** Medium
- **Location:** REQ-MA-04, Section 3.2 (Out of Scope)
- **Issue:** REQ-MA-04 says ❌ must be added when `handleMessage()` encounters a "Temporal failure". However, the current `handleMessage()` implementation exits early and silently (no acknowledgement) when `queryWorkflowState()` throws a non-`WorkflowNotFoundError` exception (lines 299-303, `temporal-orchestrator.ts`). This early-exit path is neither acknowledged with ❌ nor listed in out-of-scope. The developer will see silence, contradicting US-35.
- **Recommendation:** Either (a) add query failures to the in-scope acknowledgement paths in REQ-MA-04 and REQ-MA-05 (operation label: `query workflow`), or (b) explicitly list "Temporal query failure" in Section 3.2 Out of Scope with a rationale. Currently both US-35 and scope are contradictory.

---

### Finding 3: "Plain-text reply" semantic is ambiguous — postPlainMessage vs replyToMessage
- **Severity:** Medium
- **Location:** REQ-MA-02, REQ-MA-05
- **Issue:** The REQ calls for a "plain-text reply in the thread" but `DiscordClient` exposes two distinct methods: `postPlainMessage(threadId, content)` which sends a new top-level thread message, and `replyToMessage(channelId, messageId, content)` which sends a Discord reply (quote-linked to the triggering message). These produce different visual representations in Discord. The REQ does not specify which is required, leaving it ambiguous for the implementer.
- **Recommendation:** Clarify whether the "reply" for REQ-MA-02 and REQ-MA-05 should use `postPlainMessage` (unthreaded, appears in sequence) or `replyToMessage` (visually linked to the triggering message). Given the stated goal of "the user knows specifically what happened", `replyToMessage` is the stronger UX choice, but must be explicit.

---

### Finding 4: REQ-MA-05 operation labels do not cover all error code paths
- **Severity:** Medium
- **Location:** REQ-MA-05
- **Issue:** REQ-MA-05 enumerates exactly three `{operation}` label values: `query workflows`, `start workflow`, and `route answer`. The current `handleMessage()` implementation contains at least two additional failure paths that will produce a ❌ reaction (if REQ-MA-04 is implemented as written): (1) `phaseDetector.detect()` failure (line 440, `temporal-orchestrator.ts`) and (2) `WorkflowExecutionAlreadyStartedError` (line 467). Neither maps to the three allowed operation labels.
- **Recommendation:** Either extend the operation label enumeration to cover `detect phase` and document that `WorkflowExecutionAlreadyStartedError` is treated as a non-error (no ❌), or tighten REQ-MA-04 to explicitly exclude these paths and confirm they remain silent.

---

### Finding 5: REQ-NF-17-01 wording is contradictory — "awaited" vs "must not add latency"
- **Severity:** Medium
- **Location:** REQ-NF-17-01
- **Issue:** The requirement states acknowledgement calls "must be awaited after the Temporal operation completes" but also that "their latency must not be added to the Temporal operation path". These two statements are in tension: if the ack calls are `await`ed sequentially after the Temporal call in the same execution context, their latency IS added to `handleMessage()`. The intent is ambiguous — fire-and-forget (no await) vs sequential-await vs parallel (Promise.all). Each has different error-handling implications for REQ-MA-06.
- **Recommendation:** Clarify the intended execution model. The most natural interpretation consistent with REQ-MA-06 (must catch ack errors) is: Temporal operation completes, then ack is `await`ed in a try/catch. Reword to: "The Temporal operation must resolve before acknowledgement is sent; acknowledgement latency does not gate Temporal correctness, but the call is awaited for error-handling purposes."

---

### Finding 6: A-19 assumption about Unicode emoji is unverified — addReaction uses message.react()
- **Severity:** Low
- **Location:** Section 3.3, Assumption A-19
- **Issue:** A-19 assumes Unicode emoji strings work with `addReaction()` without encoding. The concrete implementation (`DiscordJsClient.addReaction`, lines 247-252, `services/discord.ts`) calls `message.react(emoji)`. Discord.js `react()` accepts either a Unicode emoji string or a custom emoji string `<:name:id>`. Unicode strings do work, so A-19 is likely correct, but the assumption should reference the Discord.js documentation or be verified with a quick test, not listed as a risk-bearing assumption.
- **Recommendation:** Verify with a unit or integration test. Lower the risk rating from "Med" in R-20-01 if `addReaction` is confirmed working, or promote the verification gate to a pre-ship acceptance criterion.

---

### Finding 7: REQ-NF-17-03 deduplication relies entirely on Discord native behaviour — no spec for failure
- **Severity:** Low
- **Location:** REQ-NF-17-03
- **Issue:** The requirement delegates deduplication to "Discord's native behaviour" and states "no application-layer deduplication is required". This is acceptable for normal conditions. However, the `addReaction` implementation makes two API calls (fetch channel, fetch message, then react) — if the first call succeeds but the second call fails and retries, a duplicate reaction attempt could occur. The requirement does not specify what is acceptable if Discord's deduplication fails (e.g. custom emoji, API version change).
- **Recommendation:** Add a brief note acknowledging that Discord deduplication is relied upon and that any observable duplicate is a Discord API anomaly, not an application defect. This protects the implementer from ambiguous bug reports.

---

## Recommendation

**Need Attention**

Finding 1 (High) — the workflow ID format mismatch — is a concrete defect that will cause the acceptance criteria in REQ-MA-02 to fail on first manual test. Findings 2, 3, 4, and 5 (Medium) introduce implementability gaps that will require PM and engineering alignment during FSPEC/TSPEC authoring and could cause scope creep mid-implementation. These must be resolved before the FSPEC is authored.
