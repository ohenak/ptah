## Findings

| ID | Severity | Finding | Section ref |
|----|----------|---------|------------|
| F-01 | High | `FakeTemporalClient` has no method to simulate a `startFeatureWorkflow` return value with a specific `workflowId`. Currently `FakeTemporalClient.startFeatureWorkflow` always returns `ptah-${params.featureSlug}` (hardcoded in factories.ts line 1451). AT-MA-01, AT-MA-02, AT-MA-13, and AT-MA-14 depend on observing the exact `workflowId` passed to `addReaction` / `replyToMessage`. If `handleMessage` calls `startWorkflowForFeature` (which calls `startFeatureWorkflow` internally), the workflowId returned is the hardcoded string — tests can still assert on it, but AT-MA-02 specifies `workflowId: "ptah-test-feature"` while the AT Given says the client is configured to return that value. The AT does not match the existing fake: no `startWorkflowResult` or `startWorkflowId` injection property exists. If the implementation calls `startWorkflowForFeature` (which wraps `startFeatureWorkflow`), test authors may need to configure the fake differently. The AT setup for AT-MA-01/02/06 must be reconciled with the actual call chain before implementation begins. | AT-MA-01, AT-MA-02, AT-MA-06 |
| F-02 | High | AT-MA-11 and AT-MA-12 specify `FakeDiscordClient.addReactionError` as a per-instance error property — this property already exists in `FakeDiscordClient` (factories.ts line 626). However, `addReactionError` is a global error that causes **every** call to `addReaction` to throw, regardless of which message or emoji. AT-MA-12 requires `addReaction` to throw while `replyToMessage` succeeds. This is possible with the existing fake since `replyToMessageError` is separate. But AT-MA-11 also tests a workflow-started path where `replyToMessage` should still be called. The global `addReactionError` will suppress the reaction and then `replyToMessage` runs — the AT says "WARN logged AND `handleMessage()` returned without throwing", which is testable. However, no AT tests the dual-failure case ("both `addReaction` and `replyToMessage` throw") from FSPEC Section 6.2 / 6.3. This partial-success scenario (two WARN logs) is specified in Section 6.3 but has no corresponding AT. | Section 6.3, AT-MA-11, AT-MA-12 |
| F-03 | Medium | The ordering guarantee in AT-MA-13 is not implementable with the existing `FakeTemporalClient` / `FakeDiscordClient` infrastructure. The AT requires "Both Temporal and Discord calls are recorded with sequence numbers." Neither fake currently records a unified global call sequence. The AT describes a new test capability (cross-fake call-order recording) that must be added to the fakes before the test can be written. If this capability is not explicitly added to the TSPEC/PLAN, an implementer may skip or weaken this test. The FSPEC should note this as a required fake extension or provide an alternative approach (e.g. recording timestamps, using a shared call log). | AT-MA-13, REQ-NF-17-01 |
| F-04 | Medium | AT-MA-03 asserts `FakeDiscordClient.replyToMessageCalls` is empty for the signal-routed path. However, the existing test for `handleMessage` in the `waiting-for-user` state (temporal-orchestrator.test.ts line 616) asserts `discord.postPlainMessageCalls.toHaveLength(0)` — not `replyToMessageCalls`. The new FSPEC introduces `replyToMessage` as a distinct method from `postPlainMessage`. The existing test for no-reply on signal success currently checks `postPlainMessageCalls`, which is a different method from what AT-MA-03 checks. This is not a direct conflict, but the existing test will not cover the `replyToMessageCalls` assertion from AT-MA-03. Both assertions are needed. | AT-MA-03, Section 3.2 |
| F-05 | Medium | The FSPEC defines `handleMessage` as using `message.threadId` as the `channelId` parameter for `addReaction` and `replyToMessage` (FSPEC-MA-01, FSPEC-MA-02 Input rows). The `ThreadMessage` type has both `threadId` and `parentChannelId` fields (factories.ts line 1094–1106). Discord's `addReaction` API operates on a message within a channel; a thread IS a channel in Discord, so `threadId` is correct. However, the FSPEC never explicitly states this mapping: `channelId = message.threadId`. AT-MA-01 says `channelId: message.threadId` in the Then clause, which is correct, but the behavioral flows in Section 3 do not label the field. An implementer could mistakenly use `parentChannelId`. The FSPEC should add an explicit note that `channelId` in all acknowledgement calls is sourced from `message.threadId`, not `message.parentChannelId`. | FSPEC-MA-01, FSPEC-MA-02, FSPEC-MA-03, FSPEC-MA-04, FSPEC-MA-05, Section 3.1 |
| F-06 | Medium | The truncation AT tests (AT-MA-08, AT-MA-09) verify only that the error message portion is truncated. They do not verify the full assembled reply string length. A 200-character error message with a 20-character prefix `"Failed to start workflow: "` (26 chars) produces a reply that is 226 characters total. The AT should make clear whether the 200-character truncation applies to `err.message` alone (before formatting) or to the final reply string. BR-06 says "the error message in the reply is truncated to 200 characters" — the word "error message" is ambiguous between the raw `.message` property and the full reply content. FSPEC-MA-05 specifies truncation on the error message source, which is pre-formatting. AT-MA-08/09 are consistent with this but an implementer could apply the limit to the full reply string instead. | AT-MA-08, AT-MA-09, BR-06, FSPEC-MA-05 |
| F-07 | Low | AT-MA-10 tests a non-Error thrown value (`"temporal unreachable"` as a string). The Then clause asserts `content === "Failed to start workflow: temporal unreachable"`. The FSPEC specifies `String(err)` for non-Error values (FSPEC-MA-05). For a thrown string `"temporal unreachable"`, `String("temporal unreachable")` evaluates to `"temporal unreachable"`. This is correct. However, no AT tests a non-Error, non-string thrown value (e.g. a number, `null`, `undefined`, or a plain object). These are edge cases for `String(err)` and could reveal implementation bugs. A finding for completeness. | AT-MA-10, FSPEC-MA-05 |
| F-08 | Low | AT-MA-15 specifies `FakeDiscordClient.replyToMessageCalls` is empty when `queryWorkflowState` throws a non-`WorkflowNotFoundError`. The existing test for this path (temporal-orchestrator.test.ts line 1022) asserts `discord.postPlainMessageCalls.toHaveLength(0)`. Because `postPlainMessage` and `replyToMessage` are distinct methods, both assertions are needed to fully cover the no-acknowledgement contract. The existing test does not yet include a `replyToMessageCalls` assertion, meaning AT-MA-15 adds a genuinely new assertion that must be written. This is good, just worth noting the existing test will need to be updated or the AT written as a separate test. | AT-MA-15, Section 2.2 |
| F-09 | Low | The FSPEC does not specify whether `addReaction` and `replyToMessage` are called sequentially (await one, then await the other) or whether they can be called concurrently (e.g. `Promise.all`). FSPEC Section 3.1 shows them as sequential in the flow diagram. FSPEC-MA-01 Constraints says "Called after `startFeatureWorkflow()` resolves; not before" but says nothing about the relative ordering of `addReaction` vs `replyToMessage` calls. AT-MA-13 checks that both acknowledgement calls follow the Temporal call, but does not assert that `addReaction` precedes `replyToMessage`. BR-02 notes "at most once per `(messageId, emoji)` pair" but not ordering. If `Promise.all` is used, test AT-MA-12 (addReaction fail does not suppress replyToMessage) remains valid either way, but an implementer writing concurrent calls could still pass all ATs. The FSPEC should state whether sequential execution is required or if concurrent is acceptable. | FSPEC-MA-01, FSPEC-MA-02, FSPEC-MA-06 Independence, AT-MA-13 |

## Questions

| ID | Question |
|----|---------|
| Q-01 | AT-MA-01/02 specify `FakeTemporalClient.startWorkflow` configured to resolve with `{ workflowId: "ptah-test-feature" }`. The existing `FakeTemporalClient.startFeatureWorkflow` does not accept a configured return value — it derives the workflow ID from `params.featureSlug`. Is the intent that the TSPEC will add a `startWorkflowIdOverride` or similar injection property to `FakeTemporalClient`, or should test authors rely on the derived ID (by setting `featureSlug` to `"test-feature"` to produce `"ptah-test-feature"`)? |
| Q-02 | AT-MA-13 requires cross-fake call-order tracking via sequence numbers. Is this a new capability expected in the TSPEC (requiring `FakeTemporalClient` and `FakeDiscordClient` to share a call sequence counter), or should this be implemented as a lighter-weight alternative (e.g. recording call timestamps, or restructuring the test to validate indirectly)? |
| Q-03 | Section 6.3 describes a "both fail" partial-success scenario (both `addReaction` and `replyToMessage` throw) with two WARN log entries. This scenario is not covered by any AT. Is it expected to be added, or is it considered sufficiently covered by AT-MA-11 and AT-MA-12 individually? |
| Q-04 | FSPEC-MA-05 specifies UTF-8 multi-byte truncation safety ("truncation is applied to the string as a sequence of characters, not bytes"). No AT exercises a multi-byte character boundary. Is this an intentional omission (the JS `String.prototype.slice` is character-safe by default), or should an AT cover it explicitly for documentation purposes? |

## Positive Observations

- The behavioral flows in Section 3 are exceptionally precise and directly translatable to test cases. Each branch, each ack call, and each error path is diagrammed separately — no ambiguity about what to test.
- The explicit out-of-scope list in Section 2.2 and Section 7 is valuable for test authors: it prevents false test coverage for paths that must NOT be implemented. AT-MA-15 correctly validates one of these exclusions.
- AT-MA-03 explicitly asserts both the presence of the reaction AND the absence of a reply — the "must NOT" negative assertion is present, which is a strong testability signal.
- FSPEC-MA-06 Independence row ("Each acknowledgement call is wrapped independently") directly enables AT-MA-12 and removes ambiguity about whether the two calls share error handling.
- The boundary-condition tests (AT-MA-08 for exactly 200, AT-MA-09 for 201) are exactly the right level of precision — off-by-one errors in truncation are a common source of defects.
- The FakeDiscordClient already has `addReactionCalls`, `addReactionError`, `replyToMessageCalls`, `replyToMessageError` — all the infrastructure needed for AT-MA-01 through AT-MA-12 is pre-built.
- BR-03 (✅ and ❌ mutually exclusive per invocation) is stated as a business rule, making it testable without ambiguity.

## Recommendation

**Need Attention**

> F-01 (High): The AT setup for AT-MA-01/02/06 must be reconciled with the `FakeTemporalClient.startFeatureWorkflow` call chain before implementation begins — either by adding a `startWorkflowIdOverride` injection property to the fake, or by updating the AT Given clauses to describe slug-based ID derivation rather than a configured return value.
>
> F-02 (High): The dual-failure scenario (both `addReaction` and `replyToMessage` throw) is behaviorally specified in Section 6.2/6.3 but has no acceptance test. Add AT-MA-16 covering this path.
>
> F-03 (Medium): AT-MA-13 requires a cross-fake sequence-number infrastructure that does not yet exist. The FSPEC must either describe what fake extension is needed, or replace the sequence-number approach with a testable alternative (e.g. a shared ordered call log).
>
> F-04 (Medium): Clarify whether `replyToMessageCalls` assertions replace or supplement existing `postPlainMessageCalls` assertions in the signal-routed no-reply tests.
>
> F-05 (Medium): Add an explicit statement in FSPEC-MA-01 through FSPEC-MA-05 that `channelId` is sourced from `message.threadId`.
